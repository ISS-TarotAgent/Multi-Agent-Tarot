## Part 5 — Challenges and Technical Decisions (7:30 – 9:00)

**[Screen]** Stay on CI or switch back to the architecture diagram

**[Voiceover]**

> "Let me walk through three real decisions we made during development.

> **First — why we moved to LLM-based security instead of pure rule engines.**
> 
> Our initial approach used regex and keyword matching. It worked for obvious cases but missed a lot in Chinese — indirect emotional expressions, rephrased injection attempts, and anything that didn't match our keyword list exactly. We switched to LLM semantic evaluation for all three security checkpoints. But we kept the rule engine as a fallback: if the LLM call fails or times out, rules take over automatically. That way we never leave a checkpoint unguarded.

> **Second — why Clarifier runs in two phases instead of one.**
> 
> If we put everything into a single prompt — intent recognition, question generation, and answer synthesis — the prompt becomes too complex and the output format breaks unpredictably. Splitting into two phases, each with a single focused responsibility, gives us stable, structured outputs at each step and makes it easy to test each phase independently.

> **Third — a CI bug we actually hit.**
> 
> Our integration test conftest used a fixture called `free_tcp_port_factory` to spin up a local Docker container for PostgreSQL. That fixture comes from a third-party plugin we hadn't installed. In CI, pytest failed at the collection stage — before running a single test. The fix was simple: CI already provides a PostgreSQL service container and sets `DATABASE_URL` in the environment. We rewrote the fixture to just read that variable and skip if it's not set. Removed the Docker spin-up logic entirely."
