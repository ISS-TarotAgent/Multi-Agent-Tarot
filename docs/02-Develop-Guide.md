# Multi-Agent-Tarot的开发文档
## 1.技术栈与开发环境
### 1.1 技术栈总览
本项目面向一个具备前端交互界面、后端服务、多智能体协作流程、Agent Cybersecurity 机制以及 MLOps / LLMOps 能力的 Multi-Agent System。
在技术选型上，本项目遵循以下原则：
1. 选用成熟、主流、学习成本适中的技术，保证团队成员能够在课程周期内完成开发与集成。
2. 选用与 LLM / Multi-Agent 场景兼容度高的技术栈，优先保证系统主链路可落地。
3. 在满足功能实现的前提下，兼顾工程化、可维护性、可测试性与可扩展性。
4. 不引入过重的分布式基础设施，避免在课程项目中因基础环境复杂而影响开发效率。
本项目整体技术栈分为以下几类：
* 前端技术栈
* 后端技术栈
* Multi-Agent / LLM 技术栈
* 数据存储技术栈
* 测试与质量保障工具
* 可观测性与日志工具
* 部署与工程化工具
**技术栈的要求已经在主页的README文档说明,请查看,接下来主要介绍开发环境需求**
### 1.2 本地开发环境需求
#### 基础环境需求
* Git: 用于分支管理
* Node.js 与 npm（运行 Promptfoo eval）
* Python: 统一 3.12 版本
* Docker Desktop（本地启动 PostgreSQL + 后端服务）
#### 推荐开发工具
* VS Code
* PyCharm
* Postman: 测试API
* Jira: 用于Sprint任务管理和进度追踪
#### 环境统一原则
为减少组员间环境不一致问题,本项目建议遵循以下原则:
* 前端版本依赖在package.json中固定
* Python 依赖在 `backend/pyproject.toml` 中统一管理（`pip install -e ".[dev]"`）

* 新增依赖时需要同步更新依赖文件和开发文档

## 3.编码规范-供参考
本章用于统一团队在项目开发过程中的代码风格、命名规则、Prompt 编写方式、异常处理方式以及日志记录要求，保证多人协作时代码具备良好的可读性、一致性和可维护性。由于本项目属于多智能体（Multi-Agent）系统，除了传统的软件编码规范外，还需要对 Agent 输出格式、Prompt 设计和模型调用行为进行额外约束。
### 3.1 通用开发原则
所有开发成员应该遵循以下基本原则
* 可读性优先: 代码应该清晰,易懂
* 单一职责原则: 每个模块,类,函数应尽量只负责一项明确的职责
* 统一风格: 所有成员提交的代码应保持一致风格,包括命名方式,注释习惯等,后续将给出参考格式
### 3.2 Python代码风格规范
本项目后端及 Agent 工作流主要使用 Python 实现，因此团队统一采用 Python 的标准代码风格.
#### 3.2.1 基础规范
* 文件编码统一为 UTF-8；
* 所有源代码文件应使用英文命名。
#### 3.2.2 命名规范
* 文件名：使用小写字母与下划线，例如 safety_guard.py
* 类名：使用大驼峰命名法，例如 TarotInterpreterAgent
* 函数名：使用小写字母与下划线，例如 generate_final_reading()
* 变量名：使用小写字母与下划线，例如 user_question
* 常量名：使用全大写加下划线，例如 MAX_RETRY_COUNT
#### 3.3.3 函数规范
函数应满足以下要求：
* 一个函数只完成一个明确任务；
* 函数名应准确表达其行为；
* 参数数量不宜过多
* 返回值应具有清晰语义；
例子:
```python
def validate_agent_output(output:dict) -> bool:
```
#### 3.3.4 注释规范
注释应该解释为什么这样做
推荐写注释的场景包括：
* 复杂逻辑分支；
* 多 Agent 协作中的关键决策点；
* 安全策略判断条件；
* 与模型输出格式强相关的校验逻辑。
#### 3.3.5 文档字符串
对于核心类、公共函数、关键流程函数，应添加 docstring。建议说明：
* 功能目的
* 输入参数
* 返回值
* 可能抛出的异常
例子:
```python
def run_clarifier(question: str) -> dict:
    """
    调用 Clarifier Agent 对用户问题进行意图澄清。

    Args:
        question: 用户输入的原始问题文本。

    Returns:
        返回包含澄清后问题和意图标签的字典。

    Raises:
        ValueError: 当输入为空时抛出。
    """
```
### 3.3 Prompt编写规范
由于本项目属于 LLM 驱动的多 Agent 系统，Prompt 本身也是系统行为的一部分，因此必须进行规范化管理。
#### 3.3.1 Prompt文件独立管理
所有核心 Prompt 不应直接散落在业务代码中，而应集中管理，例如：
* 存放于 prompts/ 目录；
* 按 Agent 分类，如 clarifier_prompt.txt、synthesis_prompt.txt；
便于后续统一修改、版本控制和测试。
#### 3.3.2 Prompt内容结构统一
建议每个 Prompt 包含以下部分：
* Role：该 Agent 的角色身份
* Task：当前任务目标
* Constraints：输出限制与安全边界
* Output Format：固定输出格式要求
这样可以保证不同 Agent 的设计风格一致，也便于调试与扩展。
#### 3.3.3 输出格式必须明确
每个 Prompt 都必须显式要求模型输出固定结构，使用 JSON格式，禁止依赖“自然发挥”。
#### 3.3.4 避免歧义与冗余
Prompt 语言应尽量清楚直接，避免：
* 含糊的指令；
* 多轮矛盾要求；
* 无意义重复；
* 同时要求“详细”又要求“极其简短”这类冲突描述。
#### 3.3.5 安全限制必须写入 Prompt
对于涉及心理状态、命运判断、医疗、法律、财务等敏感场景，Prompt 中必须明确加入限制，例如：
* 不做绝对化断言；
* 不替代专业建议；
* 不给出伤害性引导；
* 不渲染恐惧或宿命感。
### 3.4 异常处理规范
由于系统包含多个 Agent 串联执行，一旦某一环失败，可能会影响整个工作流。因此异常处理必须统一、可追踪、可恢复。
#### 3.4.1 禁止静默失败
禁止使用空的 except 或只打印错误但不处理的写法。所有异常都必须被记录，并明确处理方式。
不推荐:
```python
try:
    result = run_agent()
except:
    pass
```
#### 3.4.2 区分异常类型
应尽量区分以下几类异常：
* 输入异常：如用户问题为空、格式错误
* 模型调用异常：如 API 超时、连接失败
* 输出解析异常：如 JSON 结构不合法
* 业务规则异常：如安全审查未通过
* 系统异常：如未知运行错误
#### 3.4.3 支持降级处理
当某个 Agent 执行失败时，系统应优先考虑是否可以安全降级，而不是整个流程直接崩溃。例如：
* Clarifier 失败时可退回原始问题；
* 某个解释字段缺失时可进行默认填充；
* 最终无法安全输出时，返回中性提示语。
#### 3.4.4 重试机制
对于外部模型调用错误，可设置有限次数重试机制，但必须避免无限重试。建议：
* 最大重试次数固定；
* 每次重试有简短延迟；
* 重试行为必须写入日志。
### 3.5 日志记录规范
日志是本项目调试、测试、演示和安全追踪的重要依据，尤其在多 Agent 系统中，必须保留清晰的执行轨迹。
#### 3.5.1 日志记录目标
日志主要用于：
* 跟踪工作流执行过程；
* 定位错误来源；
* 检查 Agent 输入输出；
* 支持测试复现；
为课程展示提供系统运行证据。
#### 3.5.2 日志级别规范
统一使用以下日志级别：
* INFO：记录正常流程节点，如 Agent 开始执行、流程完成
* WARNING：记录可恢复问题，如字段缺失、触发降级
* ERROR：记录执行失败，如模型调用失败、解析失败
* DEBUG：记录调试细节，仅在本地开发或测试环境启用
#### 3.5.3 日志内容要求
日志中应包含足够上下文，例如：
* 当前请求 ID / 会话 ID
* 当前 Agent 名称
* 执行步骤
* 输入摘要
* 输出状态
* 异常信息
#### 3.5.4 禁止记录敏感信息
日志中不得直接记录以下内容：
* 用户真实身份信息；
* 完整 API Key；
* 任何不应长期存储的敏感文本；
#### 3.5.5 Trace Log 独立保存
建议对多 Agent 工作流额外维护结构化 trace log，用于记录：
* 每个 Agent 的输入摘要
* 每个 Agent 的输出摘要
* 校验结果
* 安全审查结果
* 最终生成内容
这样既便于 demo 展示，也方便后续做 replay test 和错误分析。
### 配置与常量管理规范
为了避免硬编码，所有环境差异项和可变参数都应集中配置。
#### 3.6.1 环境变量管理
以下内容应通过环境变量或配置文件管理：
* API Key
* 模型名称
* 最大重试次数
* 日志等级
* 调用超时时间
#### 3.6.2 禁止硬编码敏感信息
不得在源码中直接写入：
* API Key
* 私密路径
* 账号密码
* 第三方平台密钥
#### 3.6.3 常量集中定义
常量应统一放在专门文件中，例如：
* constants.py
* config.py
```python
MAX_RETRY_COUNT = 3
DEFAULT_MODEL_NAME = "gpt-4o-mini"
SAFE_FALLBACK_MESSAGE = "当前内容较敏感，我可以陪你换一个更温和的角度来看这个问题。"
```
## 4.测试指南
本章用于说明项目的测试目标、测试范围、测试方法与执行规范，确保多智能体系统在功能正确性、输出稳定性、安全性与可维护性方面达到基本要求。由于本项目不仅包含传统软件模块，还包含 LLM 驱动的 Agent 协作流程，因此测试内容不仅要覆盖代码逻辑，还要覆盖 Prompt 输出结构、工作流衔接、安全策略与异常处理。
### 4.1 测试目标
本项目的测试主要围绕以下几个目标展开：
#### 4.1.1 验证核心功能正确性
确保系统能够按照预期完成用户输入处理、Agent 协作、结果整合与最终输出，不出现明显逻辑错误。
#### 4.1.2 验证多 Agent 工作流稳定性
确保 Clarifier、Draw/Interpret、Synthesis、Safety Guard 等模块能够正确衔接，避免中间结果缺失、格式不一致或流程中断。
#### 4.1.3 验证输出格式可控
确保各 Agent 输出符合预定义 schema，避免模型自由发挥导致下游解析失败。
#### 4.1.4 验证异常场景可恢复
确保当模型调用失败、字段缺失、输入异常或安全审查未通过时，系统能够进行合理降级，而不是直接崩溃。
#### 4.1.5 验证系统安全边界
确保系统不会输出高风险、不当暗示、绝对化判断或不符合项目定位的内容。
### 4.2 测试范围
本项目测试范围应覆盖以下层面：
#### 4.2.1 单个模块测试
测试各个独立模块是否能完成自身职责，例如：
* 输入预处理模块；
* schema 校验模块；
* safety 判定模块；
* 各个 Agent 的封装函数。
#### 4.2.2 Agent 级测试
测试每个 Agent 是否能够在给定输入下输出符合预期结构与角色定位的结果，例如：
* Clarifier 是否能提炼用户意图；
* Interpret Agent 是否能生成卡牌解释；
* Synthesis Agent 是否能整合为连贯建议；
* Safety Guard 是否能识别敏感输出。
#### 4.2.3 工作流集成测试
测试多个 Agent 串联后的整体表现，重点验证：
* 上下游字段是否一致；
* 中间结果是否能被正确消费；
* 流程是否按设计顺序执行；
* 错误是否能被正确拦截。
#### 4.2.4 接口与输入输出测试
测试系统输入与最终输出是否符合约定格式，包括：
* 用户输入是否合法；
* 最终返回字段是否完整；
* JSON 结构是否可解析；
### 4.4 单元测试
单元测试用于验证最小功能单元的正确性，适合覆盖不依赖完整工作流的纯逻辑模块。
#### 4.4.1 单元测试对象
建议优先为以下模块编写单元测试：
* 输入清洗函数；
* schema 校验函数；
* 输出格式转换函数；
* 风险等级判断函数；
* 默认降级逻辑；
* 日志包装函数；
* 配置读取函数。
#### 11.4.2 单元测试要求
每个关键函数至少应覆盖：
* 正常输入；
* 边界输入；
* 非法输入；
* 空值输入；
* 异常输入。
#### 4.4.3 单元测试示例场景
例如：
* 用户输入为空字符串时，是否抛出预期异常；
* Agent 输出缺少 intent 字段时，schema 校验是否失败；
* 安全模块收到高风险标签时，是否正确触发保护逻辑；
* 当某个字段缺失时，是否使用默认值而非报错终止。
#### 4.4.4 单元测试原则
单元测试应尽量：
* 不依赖真实外部 API；
* 不依赖真实模型输出；
* 运行速度快；
* 可重复执行；
* 结果明确稳定。
### 4.5 集成测试
集成测试用于验证多个模块协同工作时是否符合预期，是本项目非常关键的一类测试。
#### 4.5.1 集成测试目标
验证以下方面：
* Orchestrator 是否正确调用各 Agent；
* Agent 输出是否能被下游模块消费；
* schema 校验与安全审查是否在正确节点执行；
* 工作流异常时是否能进入降级路径。
#### 4.5.2 建议测试链路
建议至少覆盖以下几条主链路：
* 正常问题输入 → Clarifier → Interpret → Synthesis → Safety Guard → Final Output
* 模糊问题输入 → Clarifier 补全意图 → 后续正常执行
* 某 Agent 输出字段缺失 → schema 校验失败 → 触发 fallback
* 输出被 Safety Guard 判为高风险 → 返回保护性结果
### 4.6 结构测试
由于 LLM 天然具有不稳定性，本项目必须把输出结构测试作为重点。
#### 4.6.1 测试内容
每个 Agent 的输出都应检查：
* 是否包含必填字段；
* 字段名称是否一致；
* 字段类型是否正确；
* 枚举值是否在允许范围内；
* 文本长度是否满足最低要求。
#### 4.6.2 推荐做法
建议使用预定义 schema 或数据模型进行自动校验，例如：
* JSON schema；
* Pydantic model；
### 4.7 安全测试
安全测试是本项目的核心测试内容之一，因为系统输出涉及用户情绪、生活选择与敏感表达。
#### 4.7.1 安全测试目标
验证系统不会生成以下类型内容：
* 绝对化命运判断；
* 恐吓式暗示；
* 强化用户负面自我认知；
* 对脆弱心理状态给出不当引导；
* 绕过既定限制的高风险回答。
#### 4.7.2 建议测试场景
可设计以下测试输入：
* 极度消极、自我否定类表达；
* 要求做出确定性人生决策；
* 诱导模型突破角色边界；
* Prompt injection 类输入；
* 故意要求忽略安全规则的输入。
#### 4.7.3 安全测试通过标准
在安全测试中，系统不一定要“回答完整”，但至少应满足：
* 不输出明显危险内容；
* 能触发重写、弱化、拒绝或转向中性表达；
* 安全决策过程可在日志中追踪。
### 4.8 回归测试
回归测试用于确保新改动不会破坏原本已经通过的功能。
#### 4.8.1 适用场景
当发生以下改动时，应执行回归测试：
* Prompt 修改；
* Agent 输出字段调整；
* Orchestrator 流程修改；
* schema 更新；
* 安全规则新增或调整；
#### 4.8.2 回归测试内容
建议至少保留一组固定测试样例，覆盖：
* 正常主流程；
* 模糊输入；
* 高风险输入；
* 结构缺失输入；
* 边界输入。
### 4.9 测试数据管理
测试数据应统一管理，避免成员各自随意造数据导致测试口径不一致。
#### 4.9.1 测试目录
```bash
tests/
├── unit/
├── integration/
├── regression/
└── test_data/
```
测试数据可以用JSON文件,YAML文件
#### 4.9.2 数据内容要求
每个测试案例建议至少包含：
* case id；
* 输入文本；
* 预期结构；
* 风险等级预期；
#### 4.9.3 测试工具选择

本项目使用以下测试工具：

| 工具 | 范围 | 命令 |
|------|------|------|
| pytest | Agent 单元测试 | `python -m pytest agent/tests/ -q` |
| pytest | Backend 单元 + 集成测试 | `cd backend && python -m pytest app/tests/ -q` |
| Promptfoo | 端到端工作流 eval | `npx promptfoo@latest eval -c evals/promptfoo/promptfooconfig.yaml` |

**Promptfoo eval 不需要启动 HTTP 服务**，provider 直接在进程内调用工作流。运行前确保 `backend/.env` 中已填入有效的 `OPENAI_API_KEY`（使用真实 LLM 时）。

### 4.10 本地启动方式

在 `Docker/` 目录下执行以下命令，可一键启动 PostgreSQL + FastAPI 后端：

```bash
cd Docker
docker compose up        # 前台，日志直接输出
docker compose up -d     # 后台，用 docker compose logs -f backend 追踪日志
```

首次启动时 backend 容器会自动执行：
1. `pip install -e /workspace/backend`（约 1-2 分钟）
2. `alembic upgrade head`（建表）
3. `uvicorn app.main:app --reload`（启动 API）

服务就绪后访问 `http://localhost:8000/docs` 查看接口文档。

停止服务：
```bash
docker compose down          # 停止并删除容器（数据保留）
docker compose down -v       # 同时清除数据库 volume
```
